# 宝宝饱饱：产品定义

> 文档状态：产品形态基线  
> 目标：统一产品本质、完整用户流程、页面形态、安全边界、系统行为与首版范围，作为原型、设计和开发的共同依据。

> 逐界面落地规格：[界面文案与结构.md](./界面文案与结构.md)，用于实现每个页面的层级、正式文案、状态、交互与跳转。
> 本地运行与验收：[README.md](./README.md) · [验收报告.md](./验收报告.md)

## 1. 产品定义

### 1.1 一句话定位

粘贴一个宝宝辅食视频，系统根据宝宝档案判断能不能吃、应该怎么改，并在厨房里一步步陪用户做完。

### 1.2 产品本质

宝宝饱饱不是普通的 AI 菜谱生成器，也不是把视频转成文字的工具。

它解决两个连续问题：

1. 视频里的辅食是否适合我家宝宝？
2. 如果适合，怎样根据宝宝的情况调整，并在真实厨房环境中做完？

产品由两项核心能力构成：

- **个体适配判断**：将面向所有人的辅食内容转换为针对一个宝宝的判断与调整方案。
- **动态执行陪伴**：把调整后的方案转换为可操作的步骤，并在缺食材、做法变化、计时和临时提问时动态更新后续任务。

### 1.3 核心价值

> 把“看起来可以学”的辅食视频，变成“我知道为什么适合，并且现在就能照着做”的宝宝专属任务。

用户最终购买的不是视频识别能力，而是对适配结论和执行过程的信任。

### 1.4 建议的首要用户

首版建议聚焦：

- 正在给约 6—24 月龄宝宝制作辅食的主要照护者；
- 经常从抖音等内容平台收藏辅食视频；
- 最常产生“我家宝宝能不能吃、怎么改、怎么做”的疑问；
- 做饭时需要低阅读负担、单手或免手操作。

月龄只是首版聚焦范围，不是适配判断的唯一依据。纠正月龄、进食准备度、质地能力、食材尝试史和特殊情况都可能改变结论。

## 2. 用户问题与使用场景

### 2.1 核心用户任务

> 当我刷到一个想给宝宝做的辅食视频时，我想快速知道它是否适合我家宝宝、需要怎样修改，并在做饭过程中得到一步步指导，从而不必自己拼凑评论、搜索答案和反复猜测。

### 2.2 典型场景

1. 用户在抖音刷到辅食视频并复制链接。
2. 用户进入宝宝饱饱，粘贴链接。
3. 系统解析视频，与宝宝档案进行对照。
4. 系统给出“可以直接做、调整后可以做、需要补充信息、暂不建议或无法可靠判断”的结论。
5. 用户查看宝宝专属的食材、做法和喂法。
6. 用户进入陪做模式，根据当前步骤实际操作。
7. 用户遇到缺少食材、份量变化或做法异常时，通过快捷按钮或语音告诉系统。
8. 系统重新计算并更新后续步骤。
9. 制作完成后记录即时进食反馈，稍后补充观察反馈。
10. 反馈进入宝宝档案，使下一次适配更准确。

### 2.3 可能存在的多人协作场景

发现视频、决定是否制作和实际下厨的人可能不是同一个人。例如妈妈完成适配后，将宝宝版本发给爸爸或老人制作。

因此长期产品需要支持分享“可继续执行的任务”，而不是只分享图片：

- 接收者可查看调整后的食材与步骤；
- 可直接进入陪做模式；
- 制作结果可回到宝宝档案；
- 分享时不暴露不必要的宝宝隐私。

该能力重要，但不阻塞单人核心闭环的首版验证。

## 3. 产品原则

### 3.1 适配结论优先于菜谱摘要

用户首先需要知道“能不能给我家宝宝做”，其次才是视频讲了什么。因此结果页的最高优先级是适配结论，不是成品图、难度或摘要。

### 3.2 不确定性必须可见

系统不得为了输出完整菜谱而悄悄补齐视频中缺失的信息。所有关键事实需要区分：

- 视频明确说明；
- 从字幕或画面识别；
- 系统推测；
- 无法确认。

如果缺失信息会改变安全性或适配结论，必须追问用户或停止给出确定结论。

### 3.3 对话组织流程，组件执行动作

陪做页不是菜谱步骤卡旁边附加一个聊天框。它是一条持续生长的烹饪会话：AI 在对话中确认食材、给出当前行动、接收现场反馈并推进流程；食材清单、参考图、计时器、完成标准和风险提示作为可操作的消息组件嵌入时间线。步骤是系统内部的流程结构，不是用户需要逐页完成的页面框架。

### 3.4 每次修改都要重新校验

食材替换、份量变化、厨具变化或步骤偏差都可能改变过敏原、含水量、成型能力、时间、质地和风险。任何实质修改都必须：

1. 重新计算相关食材；
2. 更新受影响的后续步骤；
3. 再次执行适配与安全检查；
4. 告知用户具体改变了什么。

### 3.5 不用流畅掩盖未知

无法识别、无法判断或超出产品能力时，清楚说明边界比生成一个看似完整的答案更重要。

## 4. 完整产品闭环

```text
建立宝宝基础档案
        ↓
粘贴辅食视频链接
        ↓
检查链接与解析质量
        ├─ 无法读取：解释原因，提供替代输入
        ↓
提取食材、步骤、时间、质地与缺失信息
        ↓
信息是否足够？
        ├─ 不足：针对性追问
        ↓
与宝宝档案对照并完成安全检查
        ↓
生成适配结论
        ├─ 暂不建议：解释原因，不进入常规陪做
        ├─ 无法可靠判断：说明缺口，不伪造方案
        └─ 调整后可做：生成宝宝版本并再次检查
        ↓
展示宝宝专属食材、做法与喂法
        ↓
进入 AI 陪做
        ├─ 缺食材：替换、重算、重检
        ├─ 份量变化：重算用量与步骤
        ├─ 做法异常：现场纠偏
        └─ 网络中断：保留步骤、计时与完成状态
        ↓
完成制作
        ↓
记录即时进食反馈
        ↓
稍后记录观察反馈
        ↓
更新宝宝档案与食材尝试记录
```

## 5. 信息架构与页面形态

### 5.1 首次进入：宝宝基础档案

首次进入采用三张逐步卡片。除用于个性化称呼的宝宝小名外，只询问会立即改变适配结果的信息。所有卡片同时支持选择和手动输入。

#### 卡片一：宝宝现在多大？

- 宝宝称呼（小名即可，不要求真实姓名）；
- 月龄；
- 是否早产；
- 如早产，补充纠正月龄。

#### 卡片二：宝宝有哪些不能吃的？

- 已确认过敏食材；
- 正在回避或医生要求避免的食材；
- 尚未尝试的常见食材或过敏原；
- 其他情况手动输入。

#### 卡片三：宝宝现在吃到什么阶段？

- 刚开始添加辅食；
- 泥糊状；
- 颗粒状；
- 手指食物；
- 容易干呕、噎住、拒绝某类质地；
- 其他情况手动输入。

#### 渐进式档案原则

首次不收集完整病史或大量资料。遇到具体菜谱时，再按需询问：

- 某个食材以前是否吃过；
- 是否出现过疑似反应；
- 是否能处理当前大小和质地；
- 是否存在吞咽、发育或特殊医疗情况；
- 家中有哪些厨具。

每个问题都要能说明“为什么现在需要问”。

### 5.2 首页：一个主入口、两个次入口

首页解决三种不同起点，但视觉上只突出最高频、意图最明确的主任务。

1. **主入口：粘贴辅食视频链接**。适用于“我已经刷到一个想做的视频”，占据首屏主卡，包含链接输入、剪贴板读取和“分析视频”。
2. **次入口：给宝宝安排辅食**。适用于“我不知道接下来做什么”，依据宝宝档案、食材记录和做饭节奏生成今天、未来 7 天或未来 4 周的方向。
3. **次入口：宝宝食物地图**。适用于“我想知道吃过什么、下一种试什么”，按食物类别查看事实记录，并进入单个食材的探索计划。

三个入口必须在标准手机首屏内同时可见，但不能同权：首页不重复展示最近分析；历史内容统一进入底部“记录”。只有真实存在未完成的陪做会话时，才展示轻量恢复入口。

替代输入可包括：

- 上传用户有权使用的视频；
- 上传关键截图；
- 手动输入菜名、食材或做法。

### 5.3 辅食计划

计划不是每天必须照做的日历，也不是另一个开放式搜索框。它先收集会改变方案的三个条件：时间范围、每周顿数与备菜节奏、新食材安排。

- **今天**：输出一顿可立即执行的方案；
- **未来 7 天**：默认推荐，输出 3、5 或 7 顿菜单、集中采购与备菜安排；
- **未来 4 周**：只给每周重点和食材方向，不排满每天。

每周方案限制同时引入的新主食材数量，并把每道菜连接到候选视频或宝宝版本。用户反馈后允许调整后续安排。

#### 计划与视频的闭环

计划不直接假设某一道菜已经有可信做法。每一顿先形成“内容需求”，至少包含目标食材、质地、做法和时间，再进入候选视频检索：

1. 从有权使用的候选库或已接入的内容源检索视频；
2. 展示视频与计划匹配的原因，但不把“匹配”写成“适合宝宝”；
3. 用户选择视频后创建标准 `analysisId`，复用“视频解析 → 档案对照 → 宝宝版本 → 陪做 → 反馈”链路；
4. 无候选时允许用户粘贴自己找到的链接或上传有权使用的视频；
5. 分析和进食反馈回写记录，成为下一次计划的输入。

内容源能力必须与后续分析解耦。当前实现使用本地候选库，并通过 `/api/plan-videos` 暴露检索契约；未来接入经过授权的抖音内容源时，只扩展检索提供方，不绕过平台权限、不假设可以抓取全站，也不改动分析和陪做流程。

### 5.4 宝宝食物地图

食物地图记录“发生过什么”，不表达医学意义上的过敏排除结论。核心层级为：

1. 全局进度：已有记录、当前成长站、下一种计划尝试；
2. 月龄成长路线：以 `6 月龄、7—8 月龄、9—10 月龄、11—12 月龄、13—18 月龄、19—24 月龄` 六个章节组织记录；
3. 食物路径节点：使用曲线路径串联“已点亮、下一种、待探索”三种状态，每站放置 4 个食物形象；
4. 单个食材详情：为什么现在安排、相关菜谱、制作记录、即时反馈与后续观察。

月龄章节只是帮助家长理解宝宝的食物经历，不是固定食材准入表。节点顺序由宝宝档案、实际记录和计划共同决定，不表达“某种食物只能在某个月龄尝试”。首版共预留 `24` 个独立食物形象位。

产品用“食材探索计划”替代“排敏计划”表述；出现异常时停止普通流程，并提示遵循既有专业建议或寻求专业帮助。

### 5.5 分析过程页

系统以真实阶段展示进度，不使用虚假的固定百分比：

1. 正在读取视频；
2. 正在识别食材和步骤；
3. 正在与宝宝档案对照；
4. 正在生成宝宝版本；
5. 正在进行最终检查。

如果系统需要用户补充信息，应立即暂停并提问，不让用户等待完整分析后再返工。

### 5.6 视频分析结果页

#### A. 原视频

- 在顶部保留原视频或来源卡片；
- 展示原作者和来源；
- 允许回看关键片段；
- 无法嵌入时展示封面和“打开原视频”。

#### B. 宝宝适配结论

结论必须属于以下五种之一：

| 结论 | 含义 | 下一步 |
| --- | --- | --- |
| 可以直接做 | 未发现需要调整的问题 | 可进入宝宝版本 |
| 调整后可以做 | 修改后可尝试 | 展示修改并重新校验 |
| 需要补充信息 | 缺少会改变判断的信息 | 回答后重新判断 |
| 暂不建议 | 存在明确不适龄、已知回避或其他风险 | 解释原因，不进入常规陪做 |
| 无法可靠判断 | 视频或输入质量不足 | 补充材料或放弃本次解析 |

结论卡必须回答：

- 为什么得出这个结论；
- 使用了宝宝档案中的哪些信息；
- 视频中发现了什么；
- 哪些内容已被调整；
- 哪些内容仍然不确定。

#### C. 关键调整

在详细菜谱前，集中展示本次最重要的变化。例如：

- 去掉某种调味料；
- 将某个食材替换为已尝试食材；
- 延长烹饪时间；
- 调整切分尺寸和成品质地；
- 首次只建议少量尝试。

每项调整支持对照：

```text
原视频：面条煮 4 分钟
宝宝版本：煮 6 分钟并剪短
调整原因：匹配宝宝当前的质地处理能力
```

#### D. 食材

按主料、辅料、调味料和可选项分组。每项显示：

- 名称与数量；
- 信息来源和置信状态；
- 宝宝适配状态；
- 是否已经尝试；
- 是否为已知回避食材；
- 家里是否已有；
- 建议替换项。

食材状态建议统一为：

- 可以使用；
- 尚未尝试；
- 建议替换；
- 需要回避；
- 无法确认。

#### E. 份量调整

不采用普通菜谱的简单倍数换算。用户选择：

- 只做宝宝份；
- 宝宝份加成人份；
- 宝宝数量；
- 少量尝鲜或正常一餐。

系统重新计算食材、容器、烹饪时间和后续步骤。盐、糖、刺激性调味料和风险食材不能随份量机械倍增。

#### F. 做法

结果页展示完整步骤预览。每一步包含：

- 步骤编号与标题；
- 当前动作；
- 用量、时间、火候和完成标准；
- 宝宝专属调整；
- 技巧提示；
- 风险与不确定信息；
- 对应的原视频片段。

#### G. 喂法

- 本次建议份量；
- 成品应该达到的软硬度和大小；
- 首次尝试方式；
- 喂食时需要观察什么；
- 剩余食物的处理方式；
- 出现异常时应停止普通流程并寻求何种帮助。

#### H. 页面操作

固定底部主按钮：

> **按宝宝版本开始做**

保存和分享作为次级操作，不与主任务争夺注意力。

### 5.5 AI 对话式陪做页

陪做页是一段由 AI 推进、由用户现场反馈共同决定走向的持续烹饪会话。对话本身就是执行系统，不再拆成“步骤卡 + 附属聊天区”。

#### 顶部区域

- 返回与退出；
- 轻量总体进度条，不强调“第几页”；
- “保持语音”开关；
- 菜谱名称与 Mock/运行状态。

#### 持续对话时间线

系统按发生顺序保留食材确认、用户回答、当前行动、调整结果和计时状态。AI 每次只推送当前需要处理的行动，但历史上下文仍可回看：

```text
AI：这些食材都准备好了吗？
[都准备好了] [没有西蓝花] [没有鲜虾]

用户：没有西蓝花

AI：西蓝花是可选蔬菜。这次可以不放，我已经同步调整后面的煮面步骤。

AI：当前行动——处理鲜虾和蔬菜
    [切碎程度参考]
    做到什么算完成：食材洗净并分别放好

[已经达到这个状态] [我遇到问题]
```

#### 对话中的原生消息组件

- 食材清单与缺少食材选择；
- 当前行动、完成状态与宝宝专属调整；
- 图片或短视频参考；
- 运行中的计时器；
- 调整结果和对后续流程的影响；
- 风险中断与处置提示。

#### 情境快捷回答

根据当前步骤动态展示：

- 已经达到这个状态；
- 我没有这个；
- 帮我换一种；
- 这个宝宝能吃吗；
- 实际情况和视频不一样；
- 拍照帮我看看；
- 还没有达到；
- 出现异常。

#### 计时器

- 只有用户确认动作已经发生后才启动；
- 可同时存在多个计时器；
- 使用目标结束时间计算剩余时长，页面恢复后仍然准确；浏览器允许时发送通知，否则明确降级；
- 网络中断不影响本地计时；
- 计时结束后询问实际状态，而不是自动判定食物已经完成。

#### 保持语音模式

免手模式不是单纯的语音输入开关，应支持：

- 自动朗读新出现的当前行动和 AI 回复；
- 识别“下一步、重复、暂停、开始计时、还没有”等短指令；
- 用语音回答现场问题；
- 浏览器支持并获得权限时保持屏幕常亮，否则提示用户该能力不可用；
- 关键风险信息要求用户明确确认。

#### 自由输入

用户可以在任何行动中自由输入或语音提问，回答必须绑定当前宝宝、当前菜谱、当前行动和已经发生的修改。

AI 不重复输出整份菜谱，只说明当前问题、必要原因和对后续步骤的影响。

### 5.6 完成与反馈页

反馈分成两个阶段。

#### 即时制作与进食反馈

- 使用一个单页同时记录“吃了多少、接受程度、吞咽与即时情况”，不拆成三个页面；
- 三组问题都是必填选择，页面显示完成度，未完成时不可保存；
- 备注为可选，提供“面条再软些、虾滑再小些、宝宝愿意再吃”等快捷标签；
- 选择“质地有点难”后，记录会影响下次的质地适配；
- 选择“疑似异常”后，原位展开安全提示，不用等到提交后才告知用户。

#### 延迟观察反馈

- 是否出现需要关注的皮肤、呼吸或消化表现；
- 是否需要将某种食材标记为“待观察”；
- 是否需要停止继续尝试并咨询专业人员。

“宝宝不喜欢”“宝宝难以下咽”和“疑似不良反应”必须进入不同处理路径，不能统一归为“不爱吃”。

### 5.7 宝宝档案与食材记录

宝宝档案长期积累：

- 基础月龄与纠正月龄；
- 当前质地与进食阶段；
- 已确认回避食材；
- 已尝试食材；
- 尚未尝试食材；
- 待观察食材；
- 对味道和质地的接受偏好；
- 历史菜谱、实际修改与反馈。

食材不能仅有“吃过/没吃过”两个状态，建议至少支持：

```text
未记录 → 尚未尝试 → 已尝试待观察 → 接受良好
                         └→ 疑似异常待确认
                         └→ 已确认回避
```

## 6. 多种新食材与反馈归因

如果一份菜谱同时包含多个宝宝从未尝试过的食材，尤其是多个常见过敏原，出现异常后将难以判断来源。

系统应：

1. 自动识别本次菜谱中的未尝试食材；
2. 显示数量和具体名称；
3. 建议拆分尝试，或将其中部分替换为已接受食材；
4. 如果用户仍然继续，明确说明后续反馈可能无法归因；
5. 不在制作结束后直接将全部食材标记为“安全吃过”。

## 7. AI、前端与确定性能力的分工

### 7.1 AI 负责

- 理解视频、字幕、画面和用户自然语言；
- 将非结构化做法整理成结构化菜谱；
- 识别信息缺口并提出最少必要问题；
- 根据宝宝档案生成调整建议；
- 解释判断原因；
- 理解厨房现场变化并提出方案。

以上能力属于 AI 服务，不属于当前前端实现范围。前端只消费约定好的结构化结果，不直接调用模型，也不在页面代码中拼接 Prompt。

### 7.2 规则系统负责

- 已知回避食材命中；
- 食材尝试状态检查；
- 多个新食材提醒；
- 明确的年龄、质地和吞咽风险检查；
- 结论等级约束；
- 替换或调整后的再次检查；
- 需要中断普通流程的高风险状态。

### 7.3 确定性工具负责

- 份量与单位换算；
- 采购清单；
- 步骤状态；
- 本地计时、浏览器通知与恢复；
- 数据保存与恢复；
- 已完成任务和反馈记录。

原则：能由确定性工具可靠完成的任务，不交给 LLM 临场生成。

### 7.4 当前前端负责

- 页面、导航、表单和 Design System；
- 宝宝档案、食材记录、菜谱结果和陪做状态的展示与编辑；
- 本地计时器、浏览器能力检测、暂停与恢复；
- 麦克风、相机和文件选择的权限与采集界面，但不负责语音识别和图片理解；
- API Client、请求取消、重试、超时和错误展示；
- AI 返回数据的运行时校验；
- 用 Mock 数据覆盖全部正常、追问、阻断、失败和重算状态；
- 在 AI 服务未完成时，前端仍可独立演示完整产品闭环。

### 7.5 AI 同学负责

- 视频、字幕、语音和画面解析；
- 模型选择、Prompt、工具调用和推理流程；
- 适配结论、修改原因、追问和对话内容的生成；
- AI 侧安全策略、置信度和证据映射；
- 流式或异步任务服务；
- 按共同 Schema 返回稳定、可版本化的数据；
- 提供可区分、可重试的错误码，而不是只返回自然语言错误。

### 7.6 双方共同维护的接口边界

前端和 AI 服务只通过版本化契约协作。最少需要以下能力：

```ts
type AnalysisStatus =
  | 'queued'
  | 'reading_source'
  | 'extracting'
  | 'needs_input'
  | 'adapting'
  | 'validating'
  | 'ready'
  | 'blocked'
  | 'failed';

interface AiGateway {
  createAnalysis(input: VideoSourceInput): Promise<{ analysisId: string }>;
  getAnalysis(id: string): Promise<AnalysisSnapshot>;
  answerClarification(id: string, answers: ClarificationAnswer[]): Promise<void>;
  reviseRecipe(id: string, change: RecipeChange): Promise<AdaptedRecipe>;
  sendCookingMessage(sessionId: string, input: CookingInput): Promise<CookingReply>;
}
```

实现要求：

- 前端先提供 `MockAiGateway`，覆盖完整产品状态；
- AI 同学后续提供 `RemoteAiGateway`，不得侵入页面组件；
- 所有响应使用共享 TypeScript 类型和 Zod Schema 校验；
- UI 只依赖业务状态，不出现模型名、Prompt 或供应商概念；
- 契约变更必须升级版本或提供兼容层，不能直接让旧页面失效。

### 7.7 从 Mock 切换到真实数据的鲁棒性门槛

当前实现完成了两份 Mock 故事的页面闭环，但“存在 `AiGateway`”不等于页面已经可以无缝接入任意真实菜谱。接入 `RemoteAiGateway` 前必须关闭以下实现差距：

本次检查的实现证据：`app/BabyBaoApp.tsx:688` 固定分析 `demoLink`；`app/BabyBaoApp.tsx:745` 结果页直接读取 `tomatoRiceAnalysis`；`app/BabyBaoApp.tsx:1111` 只在两套陪做组件间切换；`app/store.ts:75` 把步骤上限固定为 `5`；`app/BabyBaoApp.tsx:1381` 用步骤序号决定快捷问题；`app/BabyBaoApp.tsx:1435` 用两个 Mock ID 决定反馈内容；`app/local-db.ts:5` 只保存一份全局 `current` 状态。

| 当前实现事实 | 真实数据风险 | 接入门槛 |
|---|---|---|
| 分析页固定使用 `demoLink`，不从路由加载真实分析任务 | 不同链接可能显示同一份结果 | 路由只携带 `analysisId`，页面通过 Gateway 获取对应快照 |
| 结果页直接读取 `tomatoRiceAnalysis` | 任意结果路由都可能展示番茄软饭 | 页面只消费经过运行时校验的 `AnalysisSnapshot` |
| 陪做页根据两个已知菜谱 ID 切换两个组件，未知 ID 默认落入虾滑面 | 新菜谱会被错误套用旧步骤和文案 | 单一 `CookingSessionPage` 按 `sessionId` 和通用步骤 Schema 渲染 |
| 当前步骤使用数字索引，并在 Store 中固定上限 `5` | 步骤增删、重排或恢复后会指向错误行动 | 使用稳定 `stepId`；步骤数量完全来自当前菜谱修订版本 |
| 缺食材、快捷问题和调整逻辑绑定“西蓝花、胡萝卜、虾”和特定步骤序号 | 其他食材或其他菜谱无法正确分支 | 快捷操作和替换由结构化 `capabilities`、`ingredientId`、`actionKind` 驱动 |
| 反馈页只识别两份菜谱标题和建议 | 新菜谱反馈会串到旧菜谱 | 反馈从 `CookingSessionSnapshot` 读取菜谱、宝宝和本次实际修改 |
| TypeScript 类型只有编译期检查，没有真实响应的运行时校验 | 缺字段、未知枚举或旧版本响应可能直接破坏页面 | 所有远端响应使用版本化 Zod Schema；失败进入明确降级页 |
| 陪做状态只有一份全局 `current`，没有按会话隔离 | 多份菜谱、恢复或重复进入时可能串会话 | 按 `sessionId` 持久化，历史消息和计时器绑定 `stepId` |

真实数据版的最小契约：

```ts
type CookingActionKind =
  | "prepare"
  | "cut"
  | "mix"
  | "heat"
  | "simmer"
  | "add"
  | "timer"
  | "check_texture"
  | "cool"
  | "serve"
  | "wait"
  | "generic";

interface CookingStepV1 {
  id: string;
  revision: number;
  actionKind: CookingActionKind;
  title: string;
  instruction: string;
  completionCriteria?: string;
  timerSeconds?: number;
  ingredientIds: string[];
  capabilities: Array<"can_complete" | "can_extend_timer" | "can_substitute" | "can_report_risk">;
}

interface CookingSessionSnapshotV1 {
  schemaVersion: 1;
  sessionId: string;
  recipeId: string;
  recipeRevision: number;
  profileVersion: number;
  status: "preparing" | "cooking" | "paused" | "risk_interrupted" | "completed";
  currentStepId: string | null;
  steps: CookingStepV1[];
  completedStepIds: string[];
  timers: Record<string, { endAt: number | null; durationSeconds: number }>;
}
```

契约规则：

- `id` 是持久化身份，数组位置只是展示顺序；
- 页面不得通过标题关键词、食材中文名或步骤序号推断业务能力；
- `steps` 为空、`currentStepId` 不存在、必填字段缺失时不能进入普通陪做；
- 缺少 `completionCriteria` 时显示“完成标准暂未提供”，不生成假的检查标准；
- `timerSeconds` 非法或缺失时不显示计时器，不能自行猜测时间；
- 菜谱发生修订后保留已发生的对话事实，但需要重新校验受影响的未完成步骤；
- `profileVersion` 变化后，旧分析结果标记为需要重新确认，不能静默沿用；
- Mock 和 Remote 必须通过同一套契约测试，页面不能为 Mock 单独保留业务分支。

## 8. 异常与降级状态

产品不能只设计成功路径，至少需要覆盖以下状态。

### 8.1 链接无法读取

可能原因：链接失效、需要登录、内容受限、平台限制、网络问题。

处理：

- 说明真实原因，不假装正在分析；
- 支持重新粘贴；
- 支持上传用户有权使用的视频或截图；
- 支持手动输入关键内容。

### 8.2 视频信息不足

处理：

- 列出已识别与未确认内容；
- 只追问会改变结论的问题；
- 无法确认时输出“无法可靠判断”，而不是补写完整菜谱。

### 8.3 宝宝档案不足

处理：

- 解释当前缺少什么；
- 说明该信息为什么影响判断；
- 用户补充后重新执行判断。

### 8.4 用户现场做法偏离

例如面条类型不同、已经多放了水、食物没有成型。

处理：

- 先理解已经发生的事实；
- 判断是否可补救；
- 更新后续步骤；
- 如果无法安全补救，明确建议停止本次制作。

### 8.5 网络中断或退出

- 已解析菜谱、本地步骤和计时器不丢失；
- 恢复后回到最后一个已确认步骤；
- 未经用户确认，不自动将步骤标记为完成。

### 8.6 原视频被删除

- 已保存的宝宝版本仍可查看；
- 明确标记原视频当前不可访问；
- 保留来源信息和生成时间；
- 不将已生成内容伪装为仍可验证的原视频事实。

## 9. 安全与责任边界

### 9.1 产品边界

- 产品提供辅食内容整理、个体适配参考和制作陪伴；
- 不替代儿科医生、营养师或其他专业人员；
- 不诊断疾病，不确认过敏，不处理紧急医疗情况；
- 特殊疾病、明确吞咽问题或复杂医疗情况应提示用户咨询专业人员。

### 9.2 必须中断普通流程的情况

当用户描述可能存在严重不良反应或紧急危险时，应停止普通菜谱对话，提示用户停止喂食并立即寻求当地紧急医疗帮助。具体触发规则需要由合格专业人员审查，不由产品团队凭经验自行枚举完毕。

### 9.3 内容安全原则

- 不确认视频没有展示的信息；
- 不把“可能适合”写成“绝对安全”；
- 不根据单次进食自动确认食材永远安全；
- 不把用户的自由文本直接转成医学结论；
- 任何高风险判断规则在正式上线前必须经过专业审核和测试。

## 10. 内容来源、版权与隐私

### 10.1 原视频处理原则

默认采用：

- 保留原作者、平台和来源链接；
- 优先嵌入或跳转原视频，不重新公开托管；
- 只保存完成服务所需的结构化解析和宝宝版本；
- 原视频无法访问时明确标记；
- 用户上传内容时确认其有权提供和处理。

具体实现仍需根据目标平台最新规则和法律意见确认。

### 10.2 宝宝数据原则

- 只收集完成适配所必需的信息；
- 解释每类敏感信息的用途；
- 支持查看、更正和删除宝宝档案；
- 分享任务时默认隐藏不必要的健康信息；
- 不将宝宝数据用于无关推荐或训练，除非取得清晰、单独的授权。

## 11. 视觉与交互形态

视觉实现以 [设计DNA.json](./设计DNA.json) 为唯一 token 基线。参考图只用于提取色彩、留白、卡片和动效规律，不复制其中的角色、品牌图形或具体插画。

### 11.1 目标感受

> 暖黄与柔粉的低饱和配色，让用户感到被照顾、被托住，而不是进入一个冷冰冰的健康工具。

关键词：治愈、温暖、安定、柔软、可信、低压力。

治愈感主要通过以下方式建立：

- 奶油白大面积留白；
- 暖黄承担主要行动和当前进度；
- 柔粉承担情绪背景和引导区域；
- 深棕灰文字代替纯黑；
- 大圆角、细描边和低透明度暖色阴影；
- 一屏只有一个最重要的判断或动作；
- 完成、切换和计时使用轻量反馈，不使用持续吸引注意的动画。

### 11.2 核心颜色

| Token | 色值 | 用途 |
| --- | --- | --- |
| `color.primary` | `#F4C95D` | 主按钮、进度、选中态 |
| `color.secondary` | `#F3ADA0` | 柔粉背景、引导区 |
| `color.accent` | `#E98565` | 小面积强调、录音状态 |
| `surface.background` | `#FBF6ED` | 奶油白页面背景 |
| `surface.card` | `#FFFCF7` | 普通卡片 |
| `text.primary` | `#352F2C` | 主文字 |
| `text.secondary` | `#5F5651` | 辅助文字 |

暖黄和柔粉不能承载大段浅色文字。正文保持深色，确保治愈感不以可读性为代价。

### 11.3 安全颜色语义

不能让所有信息都使用同一种橙色：

- 绿色：可以使用或已完成；
- 黄色：需要调整、确认或继续观察；
- 红色：暂不建议、已知回避或需立即注意；
- 灰色：未知、未填写或无法确认。

颜色不能成为唯一的信息载体，必须同时使用文字和图标。

### 11.4 形象与装饰边界

当前阶段**不设计、不生成、不使用任何 AI 角色、宝宝角色或吉祥物**。

页面通过以下非形象元素保持治愈感：

- 柔和的圆形和有机色块；
- 食物实拍或原视频封面；
- 圆角线性图标；
- 小面积圆点、星芒和进度图形；
- 卡片之间有呼吸感的留白。

禁止使用复杂背景纹理、粒子、3D、视差和持续漂浮动画。它们会分散厨房操作注意力，也无法改变产品决策。

### 11.5 页面视觉落点

#### Onboarding

- 奶油白背景；
- 顶部可使用一块低饱和柔粉或暖黄色域；
- 中部只放当前问题卡，不使用欢迎角色；
- 选项卡使用大圆角和清楚的选中填充；
- 底部固定“继续”，并显示三步进度。

#### 首页

- 顶部以柔粉色域建立情绪感；
- 中心是一张占据主要视觉重量的暖白链接输入卡；
- 暖黄主按钮“分析视频”；
- “给宝宝安排辅食”和“宝宝食物地图”使用两张更紧凑的横向功能卡，层级必须弱于主输入卡；
- 最近记录不在首页重复出现；仅在存在未完成陪做时显示恢复入口。

#### 分析结果页

- 原视频保留可播放封面，高度约占手机内容区的五分之一，不挤压菜谱信息；
- 适配结论在视频下方用两行完成：结论状态 + 会改变执行的关键调整；
- 食材和步骤各使用一张连续列表卡，不给每个食材或步骤再套独立卡片；
- 默认不在主页展示长总结、逐条调整理由、来源标签和信息边界；这些属于可追溯数据，不应打断下厨决策；
- 喂法压缩为一行上桌检查，仅额外保留会改变风险的提醒；
- 安全状态使用语义色和文字，不用柔粉代替警告色；
- Mock 与测试边界只留在代码和技术文档，不在首页、分析、结果、陪做、反馈或设置界面显示演示按钮、徽标或提示；
- 底部固定操作区分两层：上层为并列的“分享卡片 / 保存菜单”，下层仅保留暖黄主行动“进入对话陪做”；
- 分享卡片只包含菜名、月龄阶段、份量、适配结论、食材和三条关键做法，不包含档案中的忌口、反馈和其他隐私信息；
- 分享卡片必须先预览，再支持 PNG 保存和系统分享；系统分享不可用时自动退化为图片下载；
- “保存菜单”是真实的本机持久化操作，重复保存同一菜谱只更新当前记录，不生成重复菜单。

#### 陪做页

- 背景保持奶油白，使用单一纵向对话时间线；
- AI、用户、当前行动、计时器和参考内容使用不同的消息组件层级；
- 顶部只保留暖黄轻量进度条与“保持语音”；
- 计时器嵌在触发它的行动消息中，不形成独立页面；
- 情境快捷回答位于输入框上方，同时最多展示四个；
- 高风险提醒切换到明确的红色语义，并中断普通会话推进。

#### 反馈页

- 使用大按钮和表情语义文字，但不依赖角色插画；
- “喜欢程度”“吞咽问题”“疑似异常”分成独立区块；
- 完成反馈只做短暂勾选和轻触觉，不做庆典动画。

### 11.6 动效原则

- 微交互约 `140ms`，普通切换约 `240ms`，页面级过渡不超过 `420ms`；
- 使用淡入、8px 上移或 `0.98 → 1` 的轻缩放；
- 不使用强回弹、连续抖动或大位移；
- 所有动效尊重系统 `Reduce Motion`；
- 计时和安全信息不依赖动画才能理解。

## 14. 核心指标

### 14.1 北极星结果

用户成功完成一次经过宝宝适配的辅食制作，并留下可用于后续适配的真实反馈。

### 14.2 过程指标

- 有效链接解析成功率；
- 从提交链接到得到可解释结论的时间；
- 信息不足时的追问完成率；
- 结果页进入陪做的比例；
- 陪做任务完成率；
- 中断后恢复率；
- 食材替换后的继续完成率；
- 即时反馈和延迟反馈完成率；
- 用户保存第二条视频并再次使用的比例。

### 14.3 安全与质量指标

- 关键食材和步骤漏识别率；
- 将未知信息错误表述为确定事实的比例；
- 已知回避食材漏拦截率；
- 调整后未重新检查的比例；
- 高风险反馈未进入中断流程的比例；
- 用户对“为什么这样判断”的理解度。

安全指标优先于回答长度、对话次数和表面活跃度。

## 15. 首版验收标准

产品达到首版可用，至少需要满足：

1. 用户可以在不阅读说明书的情况下完成宝宝基础档案；
2. 用户粘贴一条可读取视频后，可以看到结构化解析结果；
3. 系统能明确区分视频事实、推测和未知；
4. 系统只给出预定义的五级适配结论，并能解释原因；
5. 命中已知回避食材时不会进入“可以直接做”；
6. 存在多个未尝试食材时能够提醒并提供拆分或替换方案；
7. 用户替换关键食材后，食材、步骤、时间和结论会同步更新；
8. 用户可以按步骤完成制作，页面切出再返回后计时结果仍然准确；
9. 网络短暂中断或退出后，可以恢复到最后确认的步骤；
10. 用户能分别记录接受度、吞咽问题和疑似异常；
11. 本次反馈会正确更新食材尝试状态，而不是直接将所有食材标记为安全；
12. 无法读取或无法可靠判断时，系统能够清楚失败，不伪造完整结果。

## 16. 已固定与仍需固定的决策

### 16.1 本轮已固定

- 当前先做非 AI 前端，AI 接入由另一位同学负责；
- 前端通过版本化 `AiGateway` 与 AI 服务隔离；
- 当前不做任何角色形象或吉祥物；
- 视觉采用暖黄、柔粉、奶油白的低饱和治愈风格；
- 当前交付载体为移动端优先的 Web 应用；
- 前端先通过 Mock 覆盖完整产品状态，再接真实 AI 服务。
- 主演示固定为“10 月龄宝宝 + 宝宝虾滑面”；
- 第一阶段交付为本地可运行页面，不部署公开网址。

网页在手机浏览器中自适应全屏，在桌面浏览器中以居中的手机尺寸画布展示。当前目标是验证手机端产品效果和完整交互，不伪装成原生 App，也不承诺浏览器无法稳定提供的后台能力。

Web 端需要明确降级：浏览器进入后台后，持续运行、通知、屏幕常亮和录音能力都可能受到系统限制。计时器必须基于目标时间戳恢复，而不能只依赖前台 `setInterval`；无法获得权限时要给出明确提示。

### 16.2 仍需固定

以下问题不应在实现中临时决定：

1. 首版支持哪些视频输入方式，以及无法读取抖音链接时的默认替代路径；
2. 五级适配结论的具体规则、展示方式和哪些情况必须阻断陪做；
3. 哪些安全规则必须由专业人员审核后才能上线；
4. 哪些信息允许 AI 推测，哪些必须由用户确认；
5. 食材标准名称、过敏原和尝试状态的数据体系；
6. 原视频保存、嵌入、跳转和删除后的处理方式；
7. 延迟观察在什么时间提醒，以及用户如何关闭或升级提醒；
8. 宝宝敏感信息的保存期限、删除机制和分享默认值；
9. 语音采集是按住说、点击开始/结束，还是后续再做连续免手模式。

## 17. 前端技术基线与开源组件

### 17.1 推荐技术基线

- **React + TypeScript + Vite**：移动端优先的 Web 主体；
- **React Router**：页面路由、历史记录和分享任务入口；
- **Tailwind CSS 4 + CSS Variables**：将 Design DNA 固定成设计 Token；
- **Radix Primitives / shadcn/ui 开放代码组件**：复用可访问性、焦点管理和复杂交互行为，再按项目风格重绘；
- **Zod**：API 和本地数据运行时校验；
- **TanStack Query**：远端异步状态；
- **Zustand**：短生命周期 UI 与 CookingSession 状态；
- **IndexedDB**：宝宝档案、历史菜谱、反馈和中断恢复的本地持久化。

Vite 负责前端构建，不承担 TypeScript 类型检查；CI 中必须单独运行 `tsc --noEmit`。

### 17.2 手机画布展示规则

#### 手机浏览器

- 应用宽度为 `100%`；
- 最小高度使用 `100dvh`；
- 适配 `env(safe-area-inset-top)` 和 `env(safe-area-inset-bottom)`；
- 底部导航与操作栏不遮挡内容；
- 不禁用页面缩放，不依赖 Hover；
- 可点击目标不小于 `44px × 44px`。

#### 桌面浏览器

- 应用以完整手机设备框居中展示；
- 内容视口采用约 `390 × 828`，包含机身后的整体比例约为 `410 × 864`；
- 根据桌面可用高度等比缩放，不拉伸手机比例；
- 画布外使用安静的暖灰背景；
- 使用深色机身边框、大圆角、状态栏、灵动岛与轻阴影明确表达手机设备；
- 页面正文只在手机屏幕内部滚动，不能推动整个设备外框；
- 一级页面底部导航固定在手机屏幕底部；
- 固定底部操作必须相对于应用画布，而不是整个桌面窗口。

验收视口至少覆盖：`360 × 800`、`390 × 844`、`430 × 932` 和桌面 `1440 × 900`。

### 17.3 组件选择原则

不直接套用 Material、Ant Design 等完整视觉主题。它们会让产品快速变成通用工具风格，并增加覆盖样式的成本。

采用两层结构：

1. **项目自有视觉组件**：Button、Card、ActionMessage、QuickReplyBar、StatusBadge 等，在语义化 HTML 和 Design Token 上实现；
2. **复用行为复杂的开源能力**：导航、抽屉、视频、列表、动画、通知、录音和表单状态。

这样既能保持独特的治愈风格，又不用重新实现复杂手势和原生能力。

### 17.4 可直接采用的开源组件映射

| 产品位置 | 组件/库 | 用法 | 决策 |
| --- | --- | --- | --- |
| 页面路由 | [React Router](https://reactrouter.com/) | Onboarding、结果页、陪做页和分享任务路由 | 采用 |
| 样式与 Token | [Tailwind CSS](https://tailwindcss.com/docs/installation/using-vite) | 将 Design DNA 映射为 CSS Variables 和 Utilities | 采用，禁止页面散落任意色值 |
| 无样式交互原语 | [Radix Primitives](https://www.radix-ui.com/primitives/docs/overview/introduction) | Dialog、Tabs、Select、Progress、Radio Group | 采用，保留可访问行为并重绘视觉 |
| 开放代码组件 | [shadcn/ui](https://ui.shadcn.com/docs) | 表单、Skeleton、Sheet、Drawer、Toast 等基础代码 | 选择性复制，不套用默认主题 |
| 移动端底部抽屉 | [shadcn Drawer](https://ui.shadcn.com/docs/components/base/drawer) | 份量调整、食材替换、补充信息 | 采用并按 28px 圆角重绘 |
| 动画与 SVG 进度 | [Motion for React](https://motion.dev/docs/react) | 卡片切换、步骤过渡、进度环 | 采用，只动画 `transform` 和 `opacity` 为主 |
| 图标 | [Lucide React](https://lucide.dev/guide/packages/lucide-react) | 统一圆角线性图标 | 采用，缺失图标用项目 SVG 补充 |
| 原视频播放 | 原生 `<video>` | 结果页视频和关键步骤片段 | 优先原生能力，无法播放时回退来源卡片 |
| 图片 | 原生 `<img>` + `loading="lazy"` | 封面、食物图和加载占位 | 不为简单能力引入额外库 |
| 本地数据 | [Dexie](https://dexie.org/) | IndexedDB 中的宝宝档案、记录和反馈 | 采用 |
| 录音采集 | 原生 `MediaRecorder` | 录制语音并交给 AI 服务 | 条件采用；权限失败时允许文字输入 |
| 屏幕常亮 | Screen Wake Lock API | 陪做页免手模式 | 渐进增强，不作为核心流程前提 |
| 表单 | [React Hook Form](https://react-hook-form.com/) + [Zod](https://zod.dev/) | 三卡片档案、补充问题和反馈 | 采用 |
| 服务端状态 | [TanStack Query](https://tanstack.com/query/latest/docs/framework/react/overview) | 分析轮询、超时、重试和缓存 | 采用 |
| 本地 UI 状态 | [Zustand](https://zustand.docs.pmnd.rs/) | 陪做进度、临时输入和抽屉状态 | 采用，但不替代持久化数据库 |

### 17.5 不直接引入的组件

- 完整 UI Theme Kit：视觉覆盖成本高，且会稀释设计语言；
- Lottie 角色动画：当前没有形象需求；
- Three.js、Canvas 特效、粒子和 shader：不能改善核心任务；
- 第三方聊天 UI Kit：陪做虽然采用会话时间线，但消息包含计时器、参考图、完成状态和安全中断等领域组件，通用聊天气泡无法表达完整业务状态；
- 通用计时器组件：计时状态需要与步骤、通知和恢复机制绑定，应自行实现业务层。

### 17.6 需要自建的项目组件

这些组件视觉和业务耦合较强，应在项目内实现：

- `BbButton`：主、次、危险和文字按钮；
- `BbCard`：基础治愈风格卡片；
- `ProfileQuestionCard`：三步档案问题；
- `SuitabilityCard`：五级适配结论；
- `EvidenceLabel`：明确、识别、推测、未知；
- `IngredientRow`：食材状态、数量和替换入口；
- `AdjustmentDiff`：原视频与宝宝版本对照；
- `ConversationTimeline`：持续烹饪会话与自动定位；
- `ActionMessage`：当前行动、完成状态与宝宝版本调整；
- `QuickReplyBar`：由当前会话状态驱动的快捷回答；
- `CookingTimer`：本地计时、通知和恢复；
- `VoiceCaptureBar`：录音权限、波形状态和上传状态；
- `BottomActionBar`：固定底部主行动；
- `FeedbackSelector`：喜欢程度、吞咽和异常分区；
- `AsyncStatePanel`：加载、追问、失败、阻断和重试。

### 17.7 Mock 代码标注与隔离规范

Mock 不能散落在页面组件里，统一采用 Adapter 注入：

```text
src/
├── contracts/                 # 前端与 AI 共同的数据契约
├── services/ai/
│   ├── ai-gateway.ts          # AiGateway 接口
│   ├── mock-ai-gateway.ts     # @mock-only Mock 实现
│   └── remote-ai-gateway.ts   # 真实服务实现，占位但不伪造完成
├── mocks/
│   ├── fixtures/              # 固定业务数据
│   ├── scenarios.ts           # ready / needs_input / blocked / failed
│   └── README.md              # Mock 场景与触发方式
└── config/data-source.ts      # 根据环境变量选择 Adapter
```

环境变量统一使用：

```bash
VITE_DATA_SOURCE=mock   # 演示环境
VITE_DATA_SOURCE=remote # AI 接入环境
```

代码标注要求：

```ts
/**
 * @mock-only
 * 仅用于前端演示，返回结构必须与 AiGateway 契约一致。
 * 禁止被 remote 构建直接引用。
 */
export class MockAiGateway implements AiGateway {}
```

同时执行以下约束：

- Mock 文件、Fixture ID 和场景名称必须显式包含 `mock`；
- 页面组件只能依赖 `AiGateway`，禁止直接 import `src/mocks`；
- `mock` 环境顶部显示仅开发/演示可见的“Mock 数据”标识；
- 提供开发专用场景切换器，可直接进入五种适配结论和异常状态；
- Fixture 中加入 `meta.dataSource = 'mock'` 调试字段，但不把它当业务字段使用；
- CI 检查 `remote` 构建不得引用 `src/mocks`；
- Mock 与 Remote 响应必须经过同一份 Zod Schema；
- 禁止在组件中出现 `if (isMock)` 业务分支；差异只能存在于 Adapter 和开发工具层。

### 17.8 依赖引入门槛

每个第三方依赖在进入主分支前必须通过：

1. 与当前 React、Vite、TypeScript 和目标浏览器兼容；
2. iOS Safari、Android Chrome 和桌面 Chrome 完成核心交互验证；
3. 开源许可证允许当前产品使用；
4. 最近仍有维护，关键问题有可接受的替代方案；
5. 引入后确实减少业务复杂度，而不是只减少几行 JSX；
6. 支持无障碍和 `Reduce Motion`，或允许项目侧补齐；
7. 删除该依赖时，业务数据和 API 契约不需要重写。

首个技术 Spike 应同时验证：移动端 Drawer 手势、时间戳计时恢复、Wake Lock 降级、录音权限、视频播放和 IndexedDB 恢复。只有手机浏览器真机通过后，才锁定相关依赖版本。

### 17.9 本地交付标准

第一阶段不部署线上环境。本地交付必须满足：

- 新环境按 README 安装依赖后可启动；
- 使用单一明确命令启动开发服务器；
- 默认进入 `mock` 数据模式，不依赖 AI、数据库或其他外部服务；
- 首页可以直接进入“10 月龄宝宝 + 宝宝虾滑面”完整故事；
- 提供“一键重置演示数据”；
- 刷新页面后，宝宝档案、陪做进度和反馈仍可恢复；
- README 明确列出 Node.js 版本、安装命令、启动命令和 Mock 场景切换方式；
- `npm run build`、`tsc --noEmit` 和自动化测试能够独立执行。

## 18. 最终产品形态

宝宝饱饱的完整形态可以概括为：

> **分析页是一张有证据、有边界的宝宝专属菜谱；陪做页是一段能确认现场、推送当前行动、重算后续流程并持续语音工作的烹饪会话；宝宝档案则把每一次真实尝试变成下一次更可靠的判断。**

产品成功的标准不是生成了多少菜谱，而是用户是否理解结论、敢于按宝宝版本执行、能够完成制作，并留下真实可追踪的反馈。
