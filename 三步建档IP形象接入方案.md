# 宝宝饱饱：三步建档 IP 形象接入方案

> 状态：已按方案完成首版生成与接入  
> 范围：`/onboarding/age`、`/onboarding/avoid`、`/onboarding/stage`  
> 依据：当前页面实机、现有角色资产、`角色视觉规范.md`、`设计DNA.json`  
> 更新日期：2026-07-23

> 实现说明：14 张透明母版位于 `assets-source/profile/v1/master/`，运行资源位于 `public/illustrations/profile/v1/`；页面通过 `app/profile-illustrations.tsx` 的受限状态映射调用。

## 1. 结论

三个页面都应该使用 IP，但不应该都采用“标题上方放一张大图”的同一种方式。

| 页面 | IP 的主要职责 | 推荐形态 | 原因 |
|---|---|---|---|
| 宝宝现在多大了 | 对月龄输入给出即时、温和的视觉反馈 | 标题上方的动态年龄主视觉 | 页面任务单一，角色能强化建档的陪伴感，并让数字变化被感知 |
| 哪些食材需要避开 | 解释当前回答状态，降低填写忌口时的紧张感 | 标题区与二选一之间的状态主视觉 | “未回答 / 目前没有 / 有，需要填写”是三个清楚状态，适合由角色动作反馈 |
| 宝宝现在能稳定吃什么 | 帮用户辨认四种真实食物质地 | 四张选项卡内的角色＋质地示意图 | 第三步页面已经很长；用户需要分清的是质地，不是再看一张装饰性角色图 |

核心原则：IP 必须帮助用户理解或确认当前任务。没有信息作用的位置，不增加角色。

## 2. 当前问题

### 2.1 现状

- 第一步使用 `Baby` 线性图标，月龄变化后页面视觉没有变化；
- 第二步使用 `ShieldAlert` 线性图标，选择“目前没有”或“有，需要填写”后，顶部视觉仍然不变；
- 第三步使用 `UtensilsCrossed` 线性图标，四个阶段左侧只显示数字 `1—4`，没有呈现“顺滑、稠泥、软颗粒、软条”的实际差异；
- 三步虽然颜色不同，但视觉结构高度相似，IP 没有参与建档叙事；
- 现有 `ip/v1` 素材能表达欢迎、观察、确认、疑问等通用意图，但不能准确表达年龄区间和四种食物质地。

### 2.2 页面空间

在 `390 × 844` 视口实测：

| 页面 | 当前主卡高度 | 空间判断 |
|---|---:|---|
| 月龄 | 约 `443px` | 可将 `52px` 图标升级为 `124—140px` 主视觉，但需压缩上下间距 |
| 忌口 | 约 `418px`（未展开食材） | 可容纳 `96—112px` 状态图；展开食材后允许正常滚动 |
| 进食能力 | 约 `910px` | 不允许增加独立大头图；只替换卡片内 `40px` 数字块 |

固定底部主按钮始终优先。任何 IP 图都不能遮挡按钮、表单、错误信息或风险提示。

## 3. 统一角色定位

这只小鸡是“产品陪做向导”，不是宝宝本人，也不是医生。

它可以：

- 对用户的普通选择做视觉确认；
- 帮用户观察、区分和理解食物状态；
- 降低建档过程的陌生感和压力；
- 连接三步流程，让用户感到是同一个角色在陪伴完成档案。

它不可以：

- 用图片替代月龄、质地、忌口说明；
- 用笑脸、勾牌或绿色表达某种食材“绝对安全”；
- 暗示某个月龄必然具备某种进食能力；
- 出现在过敏、吞咽费力、医生要求等风险提示内部；
- 通过哭泣、疼痛、惊慌制造医疗化或恐吓感。

## 4. 页面一：宝宝现在多大了

### 4.1 使用位置

将当前标题上方的 `52 × 52px` `Baby` 图标替换为一张居中的 IP 主视觉：

```text
进度条
        [年龄状态 IP，132px]
        先确认基础月龄
        宝宝现在多大了？
        说明文字
        月龄输入
```

- 建议显示尺寸：标准屏 `132 × 132px`；小屏 `116 × 116px`；
- 图片与 kicker 间距：`4—8px`；
- 角色作为该页唯一主视觉，不再保留黄色问题图标；
- 图片本身透明，底色继续由白色卡片承载；
- 角色视线朝向下方月龄输入，形成自然阅读引导。

### 4.2 状态设计

不为每个月生成图片，按六个连续区间切换：

| 输入状态 | 素材 | 动作 | 不能传达的含义 |
|---|---|---|---|
| 未填写或无效 | `age-default` | 标准自然站立，轻微向下看 | 不猜测年龄 |
| `4—6 月` | `age-04-06` | 体型略小、安静坐姿，双翅放在围兜前 | 不暗示只会吃泥糊 |
| `7—8 月` | `age-07-08` | 好奇观察，身体轻微前倾 | 不暗示已经会独坐 |
| `9—10 月` | `age-09-10` | 自然站立，双翅轻扶围兜 | 不暗示已经会站立 |
| `11—12 月` | `age-11-12` | 开心挥手，动作比前一阶段主动 | 不暗示已经会走路 |
| `13—18 月` | `age-13-18` | 抱小星星或轻举成长标记 | 不暗示认知或语言水平 |
| `19—36 月` | `age-19-36` | 自信的小帮手姿态 | 不暗示会独立做饭 |

年龄图的差异只用于表现“成长感”，不能使用餐具、食物或咀嚼动作区分，避免把年龄和进食能力错误绑定。

### 4.3 交互规则

- 用户输入有效月龄后立即切换对应角色；
- 点击 `6 / 8 / 10 / 12 / 18 / 24 月`快捷项时同步切换；
- 月龄跨区间才换图，同一区间内 `+ / -` 不重复播放动效；
- 选择早产、填写纠正月龄时不改变主图；纠正月龄只通过文字和字段表达；
- 输入超出 `4—36` 时退回 `age-default`，错误文案仍是事实来源；
- 图片加载失败时退回现有 `Baby` 图标，不留空白。

### 4.4 为什么不直接复用现有素材

现有 `neutral` 或 `welcome` 可以作为开发期 fallback，但正式版本需要独立年龄图。用同一张通用小鸡替换原图标，只增加品牌感，不能给月龄输入提供反馈；用现有做饭姿势又会错误暗示能力。

## 5. 页面二：哪些食材需要避开

### 5.1 使用位置

将当前粉色 `ShieldAlert` 问题图标替换为状态主视觉，放在标题说明与二选一之间：

```text
会直接影响适配结论
有哪些食材需要避开？
说明文字
        [忌口状态 IP，104px]
[目前没有]       [有，需要填写]
```

这样图片会和回答控件直接建立关系，而不是远离交互、只承担装饰。

- 建议显示尺寸：`96—112px`，标准为 `104px`；
- 图片上下各保留 `8—12px`；
- 未选择状态居中；选择后仍在同一位置切换，不引起布局跳动；
- 展开食材列表后不复制角色到每个食材标签。

### 5.2 状态设计

| 回答状态 | 素材 | 动作与道具 | 情绪 |
|---|---|---|---|
| 尚未回答 | `avoid-unanswered` | 拿放大镜认真检查 | 专注、无焦虑 |
| 目前没有 | `avoid-none` | 轻举普通确认牌或点头 | 放松，但不表达“永远安全” |
| 有，需要填写 | `avoid-has-items` | 抱简单夹板或食材清单 | 认真确认、无恐慌 |

两个关键限制：

1. 不画牛奶、鸡蛋、花生等具体食材作为“危险物”；具体食材继续由网页标签表达；
2. `avoid-none` 只确认“当前没有明确要避开的食材”，不能画成医疗安全认证。

### 5.3 状态与操作联动

- 页面初次进入：`avoid-unanswered`；
- 点击“目前没有”：切换 `avoid-none`；
- 点击“有，需要填写”：立即切换 `avoid-has-items`，即使还未选具体食材；
- 添加或删除具体食材时不再次换图，避免高频闪动；
- 清除全部食材但仍处于“有”模式时，保持 `avoid-has-items`，由“至少选择一种”文字提示未完成；
- 从“有”切换“目前没有”触发清空确认时，弹窗中不放 IP；用户确认后再切到 `avoid-none`；
- “没吃过不等于不能吃”继续使用 `Info` 图标，不改成小鸡说话气泡。

### 5.4 可复用与新素材

- `inspect` 可作为 `avoid-unanswered` 的短期 fallback；
- `confirm` 不建议直接作为最终 `avoid-none`，因为它的绿色勾牌容易被理解为食材安全结论；
- `plan` 也不能准确表达“已确认需要回避”；
- 因此正式上线仍需三张独立忌口状态素材。

## 6. 页面三：宝宝现在能稳定吃什么

### 6.1 使用位置

这一页不增加独立大幅头图。保留标题区的紧凑结构，将每个阶段卡左侧的数字方块替换为“角色＋真实质地”的小插图：

```text
[小鸡+顺滑泥糊]  刚开始吃辅食             ○
                  目前稳定吃顺滑、没有颗粒…

[小鸡+稠泥]      能吃稠泥和细碎食物       ○
                  可以稳定处理较稠泥糊…
```

- 插图显示尺寸：`56 × 56px`，小屏最小 `52 × 52px`；
- 卡片最小高度：`76—82px`；
- 图片容器不再显示数字；四种阶段不是等级奖章；
- 插图与标题左对齐，右侧单选圆点保持现有位置；
- 选中态仍由卡片的暖黄色底色、边框和单选状态表达，图片自身不加红绿框。

### 6.2 四张质地素材

| 阶段 | 素材 | 必须看见的食物状态 | 角色动作 |
|---|---|---|---|
| 刚开始吃辅食 | `texture-puree` | 完全顺滑、表面连续、无颗粒的泥糊 | 小鸡拿小勺观察碗中泥糊 |
| 稠泥和细碎食物 | `texture-thick-puree` | 泥糊在勺上能保持形状，只有极少细碎点 | 小鸡展示盛起的稠泥 |
| 压软的小颗粒 | `texture-soft-lumps` | 清楚但柔软的小颗粒 | 小鸡用勺背轻压颗粒 |
| 软手指食物 | `texture-finger-food` | 一捏就软、容易压碎的短条 | 小鸡轻拿一条软食物，盘中保留同类示例 |

这四张图的第一主角是“食物质地”，小鸡是帮助展示质地的向导。食物区域至少占有效画面的 `35%`，缩小到 `48px` 时仍应能区分四种状态。

### 6.3 标题区处理

顶部当前的薄荷色 `UtensilsCrossed` 图标可以：

- 推荐方案：移除，减少页面高度，把视觉职责全部交给四张选项图；
- 备选方案：换成 `56—64px` 的 `inspect` 小图，但只在页面留白充足时使用。

首选移除。第三步的真正决策发生在四张卡片中，再放一张主视觉不会增加理解，反而会把首个选项推到首屏更下方。

### 6.4 风险信息边界

- “经常干呕”“吞咽看起来费力”等选项继续使用文字、选择态和 `ShieldAlert`；
- 选中吞咽费力后出现的风险提示内不放角色；
- 不能使用小鸡咳嗽、哭泣、捂嘴等画面示范风险；
- IP 只帮助辨认正常质地，不演示危险或医疗状态。

## 7. 三步连贯体验

### 7.1 视觉节奏

三步不是三张海报，而是同一角色逐步了解宝宝：

1. 第一步“认识宝宝”：角色随月龄产生温和成长反馈；
2. 第二步“核对边界”：角色从检查切换到确认或记录；
3. 第三步“理解能力”：角色退到选项内部，把视觉中心让给食物质地。

角色占比从大到小变化是有意设计：越接近具体决策，越要让真实信息而不是品牌形象成为主角。

### 7.2 统一动效

- 状态切换：旧图淡出，新图淡入，并从 `scale(0.97)` 到 `scale(1)`；
- 时长：`220ms`；
- easing：`cubic-bezier(0.22, 1, 0.36, 1)`；
- 不使用弹跳、旋转、循环呼吸或持续摆动；
- 新图预加载完成后再替换，避免透明空白；
- `prefers-reduced-motion: reduce` 时取消缩放和位移，直接替换；
- 第三步选择卡不换图，只改变卡片选中态。

### 7.3 无障碍

- 所有角色图默认 `aria-hidden="true"`，因为同一信息已有标题、输入值和状态文字；
- 不把关键含义只放在颜色或图中；
- 图片加载失败不影响选择、校验和继续按钮；
- `200%` 字体缩放时角色可缩小或让位，不能挤压正文；
- 触控区域继续由按钮和卡片承担，图片不单独可点击。

### 8.1 正式新增素材

```text
public/illustrations/profile/v1/
├── age-default.webp
├── age-04-06.webp
├── age-07-08.webp
├── age-09-10.webp
├── age-11-12.webp
├── age-13-18.webp
├── age-19-36.webp
├── avoid-unanswered.webp
├── avoid-none.webp
├── avoid-has-items.webp
├── texture-puree.webp
├── texture-thick-puree.webp
├── texture-soft-lumps.webp
└── texture-finger-food.webp
```

共 `14` 张业务状态图。建议同时保留：

- `1024 × 1024 PNG` 透明母版；
- 年龄、忌口：`320 / 640 WebP`；
- 质地卡：`160 / 320 WebP`；
- 单张运行图目标小于 `80KB`；
- 同一画布、安全边距、身体比例、头冠、围兜和胸章必须一致。

### 8.2 为什么不从现有角色设定图硬裁

现有 12 姿势适合欢迎、分析、陪做和完成状态，但没有年龄区间与质地信息。直接拼接或硬改会造成：

- 角色比例和视角不统一；
- 道具无法准确呈现四种质地；
- `confirm` 的勾牌被误读为安全认证；
- 年龄形象与做饭动作错误绑定。

正确做法是先以 `assets-source/ip/character-sheet.png` 和已确认角色母版锁定身份，再逐张生成、验收和输出透明素材。

## 9. 组件与数据映射方案

### 9.1 新组件职责

建议新增独立的档案插图组件，不把业务状态继续塞进现有通用 `CharacterIllustration`：

```ts
type ProfileIllustrationKind =
  | { type: "age"; months?: number }
  | { type: "avoid"; status: "unanswered" | "none" | "has" }
  | { type: "texture"; stage: FeedingStage };
```

组件负责：

- 将月龄、忌口、阶段映射为受限文件名；
- 输出正确的 `srcSet` 和 `sizes`；
- 处理预加载、加载失败和 reduced motion；
- 固定画布比例，避免切图时布局跳动；
- 未知值安全降级到 `age-default`、`avoid-unanswered` 或不显示。

组件不负责：

- 修改表单状态；
- 推断进食能力；
- 决定风险等级；
- 生成页面文案。

### 9.2 数据映射

```ts
function ageIntent(months?: number) {
  if (!months || months < 4 || months > 36) return "age-default";
  if (months <= 6) return "age-04-06";
  if (months <= 8) return "age-07-08";
  if (months <= 10) return "age-09-10";
  if (months <= 12) return "age-11-12";
  if (months <= 18) return "age-13-18";
  return "age-19-36";
}
```

忌口与进食阶段直接使用现有 store 枚举映射，不允许根据文案字符串匹配文件名。

## 10. 实施顺序

### 阶段 A：先确认视觉样张

1. 先做 `age-default`、`avoid-unanswered`、`texture-soft-lumps` 三张代表图；
2. 分别验证大图主视觉、状态图和 `56px` 小图三种使用场景；
3. 重点确认同一角色、透明边缘、小尺寸识别和食物质地是否准确；
4. 样张通过后再批量生成剩余 `11` 张，避免整批返工。

### 阶段 B：接入组件与第一、二步

1. 建立 `ProfileIllustration` 与文件映射；
2. 第一步接入月龄区间切换和相邻图预加载；
3. 第二步接入三种回答状态和清空确认后的切换；
4. 检查 `360 / 390 / 430px` 宽度、早产展开和长错误文案。

### 阶段 C：接入第三步

1. 用四张质地图替换数字块；
2. 在 `48 / 52 / 56px` 下检查质地区分度；
3. 检查长标题、`200%` 字体和固定底栏遮挡；
4. 确保风险提示无角色化表达。

### 阶段 D：质量验收

- 功能测试；
- 视觉回归截图；
- 图片失败降级；
- reduced motion；
- 键盘、读屏和触控区域；
- 首屏与滚动性能。

## 11. 验收标准

### 11.1 视觉

- 三步看起来由同一只小鸡陪伴，但页面层级不雷同；
- 第一步一眼能感知月龄输入已改变，且不暗示能力；
- 第二步一眼能感知当前回答状态，且不把食材画成危险物；
- 第三步不看数字也能分辨顺滑泥糊、稠泥、软颗粒和软条；
- 图片透明边缘干净，无白底、灰边、地面投影和裁切；
- 固定底部主按钮不被角色或内容遮挡。

### 11.2 功能

- 所有有效月龄正确映射到六个区间；
- 月龄无效、素材缺失和网络失败时表单仍可正常使用；
- 忌口三状态与 store 一致，确认清空前不提前改变最终状态；
- 四个质地图片与 `FeedingStage` 一一对应；
- IP 图片不成为新的输入控件，也不影响原有校验逻辑。

### 11.3 产品边界

- 没有任何图片表达“某食材绝对安全”；
- 没有把年龄图与进食能力自动绑定；
- 没有用角色替代“没吃过不等于不能吃”的说明；
- 没有在吞咽费力、过敏或专业意见提示中使用卖萌角色；
- 所有结论仍由用户填写的数据和文字规则表达。

## 12. 最终建议

优先做第三步的质地卡样张，其次做第二步状态图，最后扩展六个月龄变体。

原因不是第三步最显眼，而是它最能验证 IP 是否真正服务产品：如果 `56px` 的“角色＋食物”仍能让用户分清四种质地，说明这套角色系统具备信息价值；如果分不清，就应该改成更清楚的食物示意图，而不是为了“每页都有 IP”牺牲选择准确性。
